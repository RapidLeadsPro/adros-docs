# Client Onboarding Flows — Fresh vs Existing Ad Account

**Audience:** Engineers designing client onboarding flows on top of Adros
**Author:** Adros team
**Purpose:** Define the two onboarding journeys so an orchestration layer knows exactly who to trigger, in what order, with what handoff artifacts. Both journeys share the same backbone — only the starting phase differs.

> **Pattern (external stack):** the operator may work through an **optional** orchestration layer (OpenClaw, a custom MCP host, or another proprietary conductor). That layer routes; Adros is the marketing brain and execution plane. Research lanes (Perplexity, Manus, or custom tools) feed findings back; the conductor’s own task tracker holds cross-tool state when you use one.
>
> **Pattern (native Adros):** the operator uses **Adros CEO + Command Center**; routing and tasks still land on the same backend — there is no requirement for an external orchestration layer. See [Terminology (April 2026)](#terminology-april-2026) below.

> **See also (native Adros):** Operators can also run the same backbone through **Command Center** and **Strategy Workspace** on [app.adros.ai](https://app.adros.ai), with tasks, approvals, workspace masters, and files on the same backend. REST routes for those surfaces are indexed in [REST — Dashboard & Orchestration](/integrations/rest-dashboard). For how that coexists with external conductors, read [Integration Notes — §1](/integrations/integration-notes).

## Terminology (April 2026)

Older versions of this doc named the external orchestrator **Jarvis** and the deploy role **Hermes**. The live product uses **runtime actor IDs** (below); **Hermes** is not a current agent name.

| Actor | ID | Role (summary) |
|-------|-----|----------------|
| **Adros CEO** | `adros_ceo` | Conversational coordinator; intake; routes work into tasks |
| **Adros CMO** | `adros_cmo` | Managed Agent — audits, research synthesis, structure/strategy, optimization analysis |
| **Adros CD** | `adros_cd` | Copy + creative direction (target: full Managed Agent) |
| **Adros Deployer** | `adros_deployer` | Launch and account change execution |

**Adros Deployer** (`adros_deployer`) is the real deploy/launch owner. There is **no** legacy **Hermes** runtime and **no** standalone **Data Analyst** agent — research/audit/analysis maps to **CMO** (and deterministic tools), creative to **CD**, go-live to **Deployer**.

Below, **Conductor** means an **optional external** orchestration stack (for example OpenClaw or a custom conductor you control). Native Adros often replaces that with **CEO + Command Center** without a conductor.

## 1. The shared backbone

Every client onboarding — fresh or existing — follows this 8-step backbone. What changes between journeys is **which steps are skipped, which are audit-based vs build-based, and where Conductor enters the loop**.

```
[1] INTAKE        → who/what/where (business + goals + constraints)
[2] RESEARCH      → market, competitors, personas, keywords
[3] ANALYSIS      → for existing: audit data. for fresh: opportunity map
[4] STRUCTURE     → campaigns/ad sets/budgets
[5] CREATIVE      → copy + visuals (pattern-matched via Adros)
[6] APPROVAL      → Operator reviews, Conductor relays
[7] LAUNCH        → Adros Deployer deploys to platforms
[8] MONITOR       → daily Adros webhook → Conductor → specialist loop
```

Adros already has **9 workflow guides** encoding this backbone (see `adros/api/services/workflow_guides.py`). Conductor doesn't need to reinvent it — Conductor just calls `get_workflow_guide(type)` at the right moment and routes work accordingly.

## 2. Journey A — FRESH CLIENT (no ad account, no history)

**Trigger signals the operator gives Conductor:**
- "new client, just signed — here's their website"
- "onboard XYZ company"
- "client is starting from scratch, no ads yet"

### Flow

```
Operator ──► Conductor
         │
         │ (1) creates a tracked parent item "Onboard [client]"
         │     with subtasks linked to each phase
         │
         ├──► [INTAKE] Conductor itself
         │    - asks the operator 5 questions (or pulls from chat history if already given)
         │    - website, niche, offer, budget range, goals
         │
         ├──► [RESEARCH] Manus (research lane)
         │    Task: "Full onboarding research for [client]"
         │    Brief from Adros: GET /me/onboarding/brief?url=...
         │    Deliverables:
         │      - website crawl (about, pricing, offer, tone)
         │      - top 5 competitor teardown
         │      - market/category snapshot
         │      - persona drafts (3-4)
         │      - brand extraction (colors/fonts/logo)
         │    Returns: structured JSON + optional draft PDF brief
         │
         ├──► [INGEST] Adros (Adros CMO)
         │    Call path:
         │      - build_brand_profile(url) if Manus output missing brand bits
         │      - save_business_context(business_name, website, USP, ...)
         │      - save_persona(...) for each persona from Manus
         │      - upload_brand_asset(...) for logo/colors
         │
         ├──► [ANALYSIS] Adros (Adros CMO)
         │    Workflow: keyword_research OR market_research
         │    Output: [BusinessName]_Keyword_Research.xlsx or Market_Research.xlsx
         │
         ├──► [STRUCTURE] Adros CMO (structure planning; Adros Deployer executes approved changes)
         │    Workflow: meta_campaign_build OR google_campaign_build
         │    Output: [BusinessName]_Meta_Ads_Structure.xlsx (6 tabs)
         │            [BusinessName]_Google_Ads_Structure.xlsx
         │    NOTE: structure only — no copy, no images yet
         │
         ├──► [CREATIVE] Adros (Adros CD)
         │    Workflow: creative_only
         │    Tools: match_strategy → get_blueprint → build_creative_prompt
         │           → generate_creative (mode='bold', nano_banana_pro)
         │    Output: [BusinessName]_Meta_Ads_Copy.xlsx + images in DAM
         │
         ├──► [APPROVAL] Conductor → Operator
         │    Package: structure + copy + sample creatives
         │    Operator approves / edits / rejects in WhatsApp
         │
         ├──► [LAUNCH] Adros Deployer specialist
         │    Requires: client's ad account connected to Adros
         │    Tools: create_campaign, create_ad_set, create_ad, create_ad_creative
         │    (If ad account not ready: Conductor pauses, notifies operator,
         │     creates "connect ad account" task for client onboarding)
         │
         └──► [MONITOR] Adros daily webhook (8am SGT)
              Starts delivering alerts → Conductor → routes per alert type
```

### Who does what (Fresh)

| Phase | Specialist | Tool/System | Artifact |
|---|---|---|---|
| Intake | Conductor (direct) | WhatsApp + conductor task tracker | Intake notes on issue |
| Research | **Manus** (bridge) | Web crawl + synthesis | Research JSON + optional PDF |
| Ingest | Adros CMO | `save_business_context`, `save_persona`, `build_brand_profile` | Populated Adros DB |
| Keyword/Market research | Adros CMO | `research_keywords`, `competitor_gap`, etc. | `.xlsx` research file |
| Structure | Adros (Adros Deployer) | `meta_campaign_build` / `google_campaign_build` workflow | `.xlsx` structure file |
| Creative | Adros CD | `match_strategy` → `build_creative_prompt` → `generate_creative` | `.xlsx` copy + images |
| Approval | Conductor | Conductor task tracker + WhatsApp | Operator's decision |
| Launch | Adros (Adros Deployer) | MCP write tools (create_campaign, etc.) | Live campaigns |
| Monitor | Adros cron + webhook | Daily monitor → Conductor receiver | Webhook alerts |

## 3. Journey B — EXISTING AD ACCOUNT (already running ads)

**Trigger signals the operator gives Conductor:**
- "new client, they already have Meta ads running, audit them"
- "take over [client]'s Google Ads account"
- "audit + optimize existing ad account"

### Flow

```
Operator ──► Conductor
         │
         │ (1) creates a tracked parent item "Onboard + Audit [client]"
         │
         ├──► [INTAKE] Conductor itself
         │    Minimal: business name, ad account ID, main concern,
         │    target CPA/ROAS
         │
         ├──► [CONNECT] Adros (admin step)
         │    Client grants access to Meta/Google ad account
         │    (One-time OAuth flow through Adros dashboard)
         │    Conductor waits for "connected" signal before proceeding
         │
         ├──► [ANALYSIS] Adros (Adros CMO) ◄── KEY DIFFERENCE
         │    Workflow: meta_audit OR google_audit
         │    Tools:
         │      - list_ad_accounts, read_ad_account
         │      - list_campaigns, list_ad_sets, list_ads
         │      - list_insights (30d, 7d, breakdowns)
         │      - audit_landing_page
         │    Output: [BusinessName]_Meta_Ads_Audit.xlsx (6 tabs)
         │            Health score, top wins, top issues
         │
         ├──► [RESEARCH] Manus (research lane) — CONDITIONAL
         │    Only if audit reveals gaps:
         │      - weak competitive differentiation
         │      - missing persona understanding
         │      - unexplored angles
         │    Manus does competitor teardown + persona refresh
         │    (Skip if Adros audit already has enough context)
         │
         ├──► [INGEST] Adros CMO
         │    Backfill business_context + personas from:
         │      - audit data (what's working = real personas)
         │      - Manus research (if run)
         │      - build_brand_profile(url) for brand assets
         │
         ├──► [STRUCTURE/FIXES] Adros (Adros Deployer) ◄── DIFFERENT SHAPE
         │    Not a full build — a REALLOCATION PLAN:
         │      - pause wasters (low ROAS campaigns)
         │      - reallocate budget to winners
         │      - add missing exclusions
         │      - propose new ad sets for untapped audiences
         │    Output: audit .xlsx already contains this (Tab: Budget Reallocation)
         │
         ├──► [CREATIVE] Adros CD — CONDITIONAL
         │    Only if audit shows creative fatigue
         │    (frequency >3x, CTR declining, health score low on creative)
         │    Workflow: creative_only for winning audiences only
         │
         ├──► [APPROVAL] Conductor → Operator
         │    Package: audit summary + reallocation plan + new creatives (if any)
         │    Operator approves / edits per item
         │
         ├──► [EXECUTE] Adros Deployer specialist
         │    Tools: set_campaign_status (pause/enable),
         │           update_ad_set (budgets),
         │           create_ad_creative (new creatives),
         │           set_google_campaign_bidding_strategy
         │
         └──► [MONITOR] Adros daily webhook (continues)
              Already running — no separate "start monitor" step
```

### Who does what (Existing)

| Phase | Specialist | Tool/System | Artifact |
|---|---|---|---|
| Intake | Conductor | WhatsApp | Minimal context |
| Connect | Client + Adros | OAuth flow | Access granted |
| **Audit** | **Adros CMO** | `meta_audit` / `google_audit` workflow | Audit `.xlsx` + health score |
| Research (conditional) | Manus | Same as Journey A | Research JSON |
| Ingest | Adros CMO | Same memory tools | Populated Adros DB |
| Reallocation | Adros (Adros Deployer) | Audit `.xlsx` Tab 5/6 | Part of audit file |
| Creative refresh (conditional) | Adros CD | `creative_only` workflow | New copy + images |
| Approval | Conductor | Conductor task tracker + WhatsApp | Operator's decision |
| Execute | Adros (Adros Deployer) | MCP write tools | Updated campaigns |
| Monitor | Adros webhook | Daily loop | Alerts |

## 4. Shared artifact contract (how files flow between specialists)

This is what an external conductor needs for routing: every phase produces a **named artifact** that the next phase consumes. Naming is stable, format is stable.

| Artifact | Produced by | Consumed by | Format |
|---|---|---|---|
| `[Client]_Research.json` | Manus | Adros (ingest) | JSON schema (see §6) |
| `[Client]_Brand_Assets/` | Manus / Adros `build_brand_profile` | Adros CD | Folder in Adros DAM |
| `[Client]_Keyword_Research.xlsx` | Adros CMO | Structure phase | Excel, template-controlled |
| `[Client]_Market_Research.xlsx` | Adros CMO | Structure phase | Excel |
| `[Client]_Meta_Ads_Audit.xlsx` | Adros CMO | Adros Deployer (execute) | Excel, 6 tabs |
| `[Client]_Meta_Ads_Structure.xlsx` | Adros CMO | Adros CD | Excel, 6 tabs |
| `[Client]_Meta_Ads_Copy.xlsx` | Adros CD | Adros Deployer (launch) | Excel |
| Generated creatives (images) | Adros CD (via nano_banana_pro) | Adros Deployer (launch) | PNG in Adros DAM |
| `[Client]_Monthly_Report.pptx` | **Manus** (design lane) | Conductor → Operator | PPT, templated |

**Important:** all `.xlsx` templates are controlled by Adros via `get_output_template(template_type)`. The conductor should not hand-roll these schemas — it calls Adros, Adros returns the template spec, specialists fill it.

## 5. Conductor routing logic (what triggers what)

This is the decision tree Conductor uses when the operator says "onboard new client":

```
Operator: "onboard [client]"
  │
  ├─ Conductor asks: "Do they already have an active ad account running ads?"
  │   │
  │   ├─ NO  → Journey A (fresh)
  │   │        - Create a parent tracker item with 8 subtasks
  │   │        - assign_and_wake → Manus bridge (research)
  │   │        - On Manus completion → assign_and_wake → Adros CMO (ingest + keyword research)
  │   │        - On completion → assign_and_wake → Adros Deployer (structure)
  │   │        - On completion → assign_and_wake → Adros CD
  │   │        - On completion → notify operator for approval
  │   │        - On approval → assign_and_wake → Adros Deployer (launch)
  │   │        - Mark issue done, monitoring begins auto
  │   │
  │   └─ YES → Journey B (existing)
  │            - Create a parent tracker item with 7 subtasks
  │            - Wait for OAuth connect signal
  │            - assign_and_wake → Adros CMO (audit)
  │            - Branch: if audit flags creative fatigue
  │              → assign_and_wake → Adros CD
  │            - Branch: if audit flags weak positioning
  │              → assign_and_wake → Manus (research refresh)
  │            - Converge → notify operator for approval
  │            - On approval → assign_and_wake → Adros Deployer (execute fixes)
  │            - Monitoring already running, just resume
```

**Key rules:**
1. **Every `assign_and_wake` should advance the conductor’s task state** — avoid side channels outside the tracker
2. **Every phase completion closes its subtask + creates the next one** — Conductor reacts to tracker events (prefer event-driven updates over tight polling)
3. **The operator is only pinged at APPROVAL gates** — not for intermediate phase completions
4. **Manus is optional in Journey B** — audit data often replaces fresh research

## 6. Manus bridge contract (orchestration layer)

Manus is third-party. For it to slot into Conductor routing, implement a small bridge with this contract:

### Input (Orchestration → Research Bridge)
```json
{
  "task_id": "<task-ledger-issue-id>",
  "task_type": "onboarding_research" | "monthly_report_design",
  "client": {
    "name": "Acme Travel",
    "website": "https://example.com",
    "adros_client_id": "<client-uuid>"
  },
  "brief": "<full research prompt from Adros /me/onboarding/brief>",
  "output_schema": { /* JSON schema the research lane must conform to */ },
  "deadline": "2026-01-15T00:00:00Z"
}
```

### Output (Bridge → Conductor)
```json
{
  "task_id": "orchestration-task-id",
  "status": "complete",
  "artifacts": [
    {
      "type": "research_json",
      "path": "<gdrive link or local path>",
      "schema_version": "1.0"
    },
    {
      "type": "research_pdf",
      "path": "<gdrive link>"
    }
  ],
  "adros_ingest_call": {
    "endpoint": "POST /me/onboarding/ingest",
    "payload": { /* structured result ready for Adros */ }
  }
}
```

### Bridge responsibilities
1. Expose a wakeup endpoint `POST /bridge/manus/wake` so Conductor `assign_and_wake` works uniformly
2. Translate the conductor’s tracker row → Manus API call (or browser automation if no API)
3. Poll Manus for completion (or receive Manus webhook if supported)
4. Validate output against schema before marking complete
5. Post completion back to the conductor’s task tracker (comments + attachments)
6. Call Adros ingest endpoint with structured payload

**If Manus has no public API**, the bridge can use a supervised browser session (for example Playwright) so the external contract from Conductor’s perspective stays unchanged.

## 7. Optional onboarding helper endpoints

The paths below illustrate a common pattern for **external research lanes + Adros ingest**. Availability depends on your Adros version and deployment — **always confirm in live [OpenAPI](https://api.adros.ai/docs)** before relying on them in production.

| Method | Path | Purpose |
|---|---|---|
| GET | `/me/onboarding/brief?url=<url>&journey=fresh\|existing` | Returns structured research prompt + expected output schema |
| POST | `/me/onboarding/ingest` | Accepts research JSON, writes business_context + personas + brand assets |
| GET | `/me/reports/monthly/{client_id}/brief` | Returns deck design brief + report data for an external renderer |
| POST | `/me/reports/monthly/{client_id}/attach` | Accepts finished deck URL from a bridge |
| GET | `/me/clients/active` | Canonical client list for conductor configuration (see [Integration Notes](/integrations/integration-notes)) |

Coordinate rollout with your backend team so bridges and APIs ship together.

## 8. What orchestration layers should avoid

- Avoid hand-rolling research briefs when Adros can supply them via `/me/onboarding/brief` — Adros owns the marketing cognition.
- Avoid hand-rolling `.xlsx` schemas — call Adros `get_output_template`. Templates are versioned on the Adros side.
- Avoid putting Manus in the critical webhook-handling path. Manus should stay bridge-gated. If the bridge is down, Journey A should stall at the research step while the operator is notified — the ads platform path should not depend on the research bridge for liveness.
- Avoid polling for Manus completion in a tight loop; prefer webhooks or a slow janitor sweep.
- For Manus-shaped work, route through the conductor’s task tracker so approvals and artifacts stay auditable alongside Adros task history.

## 9. Design considerations (external orchestration)

1. **Connect-account flow for Journey B** — decide whether Conductor owns the OAuth nudge to the client, or a separate admin step does (affects UX when the client is slow to act).

2. **Monthly report cadence** — decide whether the orchestration layer schedules reports (cron) or the operator triggers them. Automatic schedules may map to an additional workflow such as `monthly_report`.

3. **Research lane: API vs browser** — bridge complexity depends on whether your research tool exposes a stable API or needs browser automation.

4. **Deck template storage** — choose where master templates live (shared drive vs Adros-served spec) based on how you want to version and distribute them.

5. **Multi-client batching** — decide whether parallel onboarding flows are allowed or serialized, based on bridge capacity and operator load.

---

## 10. TL;DR (external orchestration)

- **Two journeys, one backbone.** Fresh = build from scratch (external research → Adros structure/creative → launch). Existing = audit first (Adros audit → reallocate/fix → execute).
- **Adros encodes both journeys** in its workflow library. Conductors route work and artifacts; they should not reimplement Adros marketing logic.
- **Research lanes slot in via a bridge** that presents them like any other specialist. Bridge contract is in §6.
- **The human operator is only pinged at approval gates.** Other steps should flow agent-to-agent through the conductor’s tracker and Adros tasks.
- **Optional helper endpoints** (§7) may exist depending on version — verify in OpenAPI before building automation around them.
- **Design choices** in §9 affect how aggressively you can automate onboarding at scale.
