## 1. What Adros is

**Adros is a hybrid marketing OS:** the same FastAPI + Postgres + MCP backend powers **operator-grade dashboard workflows** and **AI agent integration**.

- **Marketing brain:** 4,022-pattern database, memory layer, creative pipeline, daily autonomous monitor. Managed Agents and MCP clients call Adros to "think like marketers" and act on Meta/Google through wrapped tools.
- **Orchestration surfaces:** [app.adros.ai](https://app.adros.ai) includes **Command Center** (client-scoped task inbox, live event timeline, approvals, blockers, comments → follow-up tasks), **Strategy Workspace** (staged strategy/creative artifacts, canonical masters, deploy request), **Files** (R2-backed, scoped storage), **Campaigns** (live metrics when accounts connected), **Adros CEO** (coordination → routed work).
- **Tasks and events are the system of record** — whether the trigger came from the UI, CEO chat, or MCP.

**Runtime actors (April 2026):**

| Actor | ID | Role |
|-------|-----|------|
| **Adros CEO** | `adros_ceo` | Conversational coordinator; intake; routes work into tracked tasks |
| **Adros CMO** | `adros_cmo` | Managed Agent — audits, research synthesis, structure/strategy, optimization analysis |
| **Adros CD** | `adros_cd` | Copy + creative direction (target: full Managed Agent) |
| **Adros Deployer** | `adros_deployer` | Launch and account change execution |

## 3. Data model (key entities)

### Users & multi-tenancy

- `users` — Supabase `auth.users` + Adros `public.users` profile
- `clients` — each user can have multiple clients; solo mode uses `client_id = NULL`
- **All data scoped** by `(user_id, client_id)` where applicable
- `client_lifecycle_state` — tracks where each client is in their journey (intake → research → creative → approved → live → optimizing)
- `client_integrations` — BYOK secrets (Cometly, WhatsApp, etc.) encrypted at rest via Fernet

### Tasks & orchestration

- `tasks` — the unit of work; has status, specialist/agent, cost, project
- `task_events` — append-only timeline: tool calls, model steps, guardrails, approvals, blockers
- `task_comments` — discussion; can optionally spawn a linked follow-up task
- `task_blockers` — structured human-required items (billing, permissions, policy)
- `projects` — grouping; each client typically has one active project
- **WebSocket** at `/ws/events/{task_id}?token=lyr_...` for real-time event streaming

### Strategy & artifacts

- Workspace stages: research → economics → structure → copy/creative → audit → deploy
- **Masters:** each stage can have a canonical "master" artifact (the current operating spec)
- **Snapshots:** workspace state serialized to R2 for audit trail
- **Deploy request:** creates an `adros_deployer` task with manifest context

### Patterns & memory

- `patterns` — 4,022 enriched ad patterns with `enrichment_metadata` + `image_generation_prompt`
- `memory_store` — business_context, personas, weekly_logs scoped by `(user_id, client_id)`
- Vector search via Qdrant for pattern similarity and memory recall

### Monitoring

- `monitoring_configs` — per-user webhook URL for daily monitor output
- Daily run at 8am SGT: 6 checks, HMAC-signed POST to webhook
- Monitor issues include `recommended_specialist` (runtime actor ID) for routing

## 6. Webhooks (monitor output)

When configured (`POST /me/monitoring/webhook`), Adros POSTs monitor results daily at 8am SGT.

**Payload structure (v1.1):**

```json
{
  "version": "1.1",
  "timestamp": "2026-04-11T00:00:00Z",
  "user_id": "...",
  "summary": { "total_issues": 3, "critical": 1, "warning": 2 },
  "issues": [
    {
      "alert_type": "creative_fatigue",
      "severity": "critical",
      "headline": "Ad frequency > 5 on 3 Meta ads",
      "summary": "...",
      "action": "Refresh creatives",
      "recommended_specialist": "adros_cd",
      "dedupe_key": "creative_fatigue|client_name|headline"
    }
  ]
}
```

**Signature:** `X-Adros-Signature: sha256=<hex>` — HMAC-SHA256 of `sort_keys=True` JSON body using your webhook secret.

**Deduping:** Same alert type + same client + same headline = same `dedupe_key`. Use this as the authoritative dedupe identifier.

**No retries** on delivery failure — design your receiver accordingly.

## 7. Contract layer

Deterministic deliverable specs keyed by `(stage, task_type)`. Each spec defines:

- `produce` — what artifacts the specialist must deliver
- `accept` — what the specialist consumes
- `recommended_specialist` — runtime actor ID (`adros_ceo`, `adros_cmo`, `adros_cd`, `adros_deployer`)
- `review_gates` — human checkpoints
- `next_stage` — where to go after completion

## 10. Daily autonomous monitor — 6 checks

| # | Check | Platform | What it looks for |
|---|-------|----------|-------------------|
| 1 | **Session health** | Adros | Missing business context, personas, or recent weekly logs |
| 2 | **Creative fatigue** | Meta | Ad frequency > threshold; flags stale creatives |
| 3 | **Search term waste** | Google | High-spend, low-conversion search terms; quality score issues |
| 4 | **RSA health** | Google | Responsive search ads below "good" strength |
| 5 | **Conversion health** | Google | Conversion tracking status and score |
| 6 | **Advantage+ audit** | Meta | Campaign configuration issues |

Each issue includes `recommended_specialist` (runtime actor ID) for routing into tasks.

## 11. Known foot-guns (integration pitfalls)

1. **Task ownership** — always verify the caller owns the task (user_id match) before mutating; the service layer relies on callers to enforce this in some paths
2. **Event ordering** — task events should be queried `ORDER BY created_at DESC` for "latest" feeds; confirm the endpoint you're using does this
3. **R2 env** — file upload endpoints require R2 credentials configured on the server; errors surface as config issues, not auth
4. **Quality track** — scorecard/guardrail endpoints return empty until migration + env flag are both live
5. **Campaign scope** — Google/Meta campaign data depends on user's connected accounts and scope selection; fallback behavior is surfaced in the UI
6. **WS auth** — WebSocket connections use token as query param (browsers cannot set Bearer header on WS); task ownership is verified server-side before streaming
7. **Monitor retries** — there are none; design your webhook receiver to be resilient
8. **Contract `recommended_specialist`** — values are runtime actor IDs (`adros_*`), not display names; legacy values like "Hermes" or "Data Analyst" are retired
9. **Bearer case** — use `Authorization: Bearer lyr_...` (capital B); some routes may be case-sensitive

## 14. Security model

- All routes require valid JWT (`lyr_...` token)
- Multi-tenant scoping enforced at query level (user_id + optional client_id)
- Client integration secrets encrypted at rest (Fernet)
- File access scoped to owning user/client
- MCP and REST use the same auth — no separate permission model
- Cross-tenant access should be verified at the service layer for any new routes
