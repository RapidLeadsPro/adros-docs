## 1. What Adros is

**Adros is a hybrid marketing OS:** the same FastAPI + Postgres + MCP stack powers **operator-grade dashboard workflows** and **external AI agents**.

- **Marketing brain:** 4,022-pattern database, memory layer, creative pipeline, daily autonomous monitor. Specialists and MCP clients call Adros to "think like marketers" and act on Meta/Google through wrapped tools (not thin generic proxies).
- **Orchestration surfaces:** `app.adros.ai` includes **Command Center** (client-scoped task inbox, live event timeline, approvals), **Strategy Workspace** (staged strategy/creative artifacts and masters), **Files** (R2-backed, scoped storage), **Campaigns** (live metrics when accounts are connected), **Adros CEO** (coordination/routing), plus chat where wired. Tasks and events are the **system of record** for automation runs ŌĆö whether the trigger came from the UI or MCP.
- **External orchestration:** Many customers still pair Adros with Jarvis, Paperclip, OpenClaw, or custom conductors. That is fine: Adros remains the **marketing capability + data plane**; the conductor owns cross-tool sequencing if you want it outside Adros.

**Mental model in one sentence:** *Adros = marketing intelligence + execution hands + **traceable tasks/events/files** on one backend. External orchestrators = optional conductors around that plane.*

### Background loops started in `main.py` lifespan
1. `token_refresh_loop` ŌĆö refreshes expiring Meta tokens daily
2. `winback_email_loop` ŌĆö hourly, runs cancellation recovery sequence
3. `reengagement_email_loop` ŌĆö hourly, credit re-engagement
4. `autonomous_monitor_loop` ŌĆö **this is the one that sends webhooks to OpenClaw**. Daily at 00:00 UTC (8am SGT), min 20h between runs

### MCP middleware chain (left to right on a request)
```
MCPAuthContextApp ŌåÆ CreditGateMCPApp ŌåÆ AutoLogMCPApp ŌåÆ StreamableHTTPASGIApp
```
- `MCPAuthContextApp` ŌĆö resolves `X-User-Token` header into a user context
- `CreditGateMCPApp` ŌĆö **hard-blocks** tool calls when user's credits Ōēż 0
- `AutoLogMCPApp` ŌĆö logs every tool call for usage + billing
- `StreamableHTTPASGIApp` ŌĆö the MCP SDK's session manager

**Implication for OpenClaw:** if a specialist's Adros call suddenly fails with a credit-exhausted error, it's this gate. Don't retry forever ŌĆö escalate.

## 3. Data model (verified)

All tables in Supabase Postgres. Source: `api/models/`.

### `users` ŌĆö `api/models/user.py`
Key fields for OpenClaw:

- `mcp_token` ŌĆö the personal MCP URL token
- `facebook_access_token`, `facebook_ad_account_ids[]`, `facebook_selected_account_ids[]`

- **`monitoring_webhook_url`** (text)
- **`monitoring_webhook_secret`** (string 64) ŌĆö HMAC-SHA256 secret

### `clients` ŌĆö `api/models/client.py`

### `business_context` ŌĆö `api/models/memory.py`
One row per (user_id, client_id). `client_id=NULL` = solo mode. Fields:
- `business_name`, `website_url`, `product_description`, `ideal_customer`

- `can_say[]`, `cannot_say[]` ŌĆö brand voice constraints

- `raw_research` (JSONB) ŌĆö Manus/Data Analyst findings land here

### `personas` ŌĆö `api/models/memory.py`
Per (user, client). Fields: `name`, `age_range`, `pain_points[]`, `desires[]`, `objections[]`, `language[]`, `platforms[]`, `persona_type` (`primary|secondary|aspirational`), `budget_weight` (% allocation).

### `benchmarks` ŌĆö `api/models/memory.py`
Competitor benchmark data. `competitor_name`, `competitor_url`, `ad_angles[]`, `hooks[]`, `offers[]`, `weaknesses[]`. This is where Manus competitor-teardown output should land.

### `weekly_logs` ŌĆö `api/models/memory.py` (the action stream)
Append-only. Every specialist action writes here:
- `action_type`, `action_description`, `reason`, `result`
- `pattern_id`, `pattern_name` ŌĆö if the action used a pattern
- `metrics_before`, `metrics_after` (JSONB)
- **Creative history fields (Meta only):** `hook_type` (7 values: `question|bold_claim|negative|curiosity|social_proof|before_after|statistic`), `visual_style` (7 values: `ugc|lifestyle|product_only|text_overlay|testimonial|animation|carousel`), `hook_text`, `cta_text`, `ad_format` (`feed|story|reel|carousel`), `creative_result` (`winner|loser|inconclusive|pending`), `creative_notes`

**Why this matters:** this is the "what has this specialist done for this client recently" memory. Any specialist waking up on a task should call `recall_weekly_log(weeks_back=4)` before acting.

### `patterns` ŌĆö `api/models/pattern.py` (the moat)
**~4,022 rows.** Fields:

- **Enrichment (Opus pass, API-visible but UI-hidden):** `image_generation_prompt`, `enrichment_metadata` (JSON), `enriched_at`
- **Scores:** `quality_score` (0-1), `performance_score` (0-1), `times_deployed`, `avg_ctr`, `avg_cpa`, `avg_roas`

### `pattern_elements` ŌĆö granular text/visual elements per pattern
`element_type` (`headline|subheadline|body|cta|image_prompt|logo`), `position`, `content_template`, `font_size_tier`, `color_role`, `max_words`.

## 7. Autonomous monitor (the webhook you receive)

**File:** `api/services/autonomous_monitor.py` (672 lines, start here)

### Schedule
- Runs daily at **00:00 UTC (8am SGT)**
- Checked every hour via `autonomous_monitor_loop`
- Minimum 20h between runs (prevents double-fires)
- Started from `main.py` lifespan as a background asyncio task

### Subscriber filter
```python
plan_tier IN ("pro", "pro_annual", "trial")
AND is_active = true
AND cancelled_at IS NULL
AND payment_failed_at IS NULL
AND (has_meta OR has_google)  # at least one ad platform connected
AND (monitoring_emails_enabled OR monitoring_webhook_url)  # at least one channel
```

### Checks (6 in order)
Each check produces 0+ issues. All wrapped in try/except ŌĆö one check failing doesn't stop the others.

1. **`session_check.session_start_check`** ŌåÆ overdue experiments + ecosystem alerts + compliance blocking
2. **`creative_fatigue_detector.detect_creative_fatigue`** ŌåÆ per Meta account (first 3) ŌåÆ `fatigued_ads` + `early_warning_ads`. Critical if frequency > 5.0
3. **`advantage_plus_detector.get_advantage_plus_campaigns`** ŌåÆ Meta Advantage+ campaigns flagged for manual review
4. **`search_term_analyzer.analyze_search_terms`** ŌåÆ Google, per customer (first 3). Issue if wasted_spend > $20. Critical if > $100. Notify operator if > $100. Also flags low quality-score keywords (>= 3 issues)
5. **`rsa_asset_monitor.analyze_rsa_assets`** ŌåÆ Google RSAs below "Good" ad strength
6. **`google_conversion_health.check_google_conversion_health`** ŌåÆ health_score 0-100, status `excellent|good|fair|poor`. Notify operator if status="poor"

### Per-issue structure (`_add_issue` in `autonomous_monitor.py` line 104)
```python
{
    "alert_type": str,            # machine-readable: "creative_fatigue" etc.
    "severity": "critical|warning|info",
    "headline": str,              # short title
    "summary": str,               # 1-2 sentence detail
    "action": str,                # what to do
    "link": str,                  # direct URL to fix
    "client_name": str,           # 'default' for solo, account_id for agency
    "recommended_specialist": str,  # "Hermes" | "Creative Director" | "Data Analyst" | "Admin"
    "notify_sam": bool,           # legacy field name ŌĆö "True if the human operator should be pinged immediately"
    "metrics": dict,              # structured metrics for routing logic
    "dedupe_key": str,            # 16-char sha256 hash (see below)
    "source_run_id": str,         # UUID correlating all issues from this run
    "timestamp": str,             # ISO8601 UTC
    # Legacy fields kept for backwards compat:
    "type": str, "title": str, "detail": str,
}
```

### Dedupe key formula
```python
dedupe_input = f"{alert_type}:{client_name}:{headline}".lower()
dedupe_key = hashlib.sha256(dedupe_input.encode()).hexdigest()[:16]
```
Same alert type + same client + same headline = same dedupe_key. OpenClaw should use this as the authoritative dedupe identifier.

### Webhook payload v1.1 (exact shape from `_send_monitoring_webhook`)
```json
{
  "event": "monitoring_alert",
  "version": "1.1",
  "source_run_id": "<uuid>",
  "user_id": "<uuid>",
  "user_email": "operator@example.com",
  "checked_at": "2026-01-15T00:00:00+00:00",
  "severity": "critical|warning|info",
  "issue_count": 12,
  "notify_sam_count": 3,
  "affected_clients": ["Acme Travel", "Acme Wellness"],
  "alert_types": ["creative_fatigue", "search_term_waste", "rsa_health"],
  "issues": [ ... array of issue objects ... ]
}
```

> **Legacy field naming:** `notify_sam_count` is the production field name and maps to the per-issue `notify_sam` flag. This is legacy naming from Adros' early development ŌĆö it means "should the human operator be pinged immediately." A future schema version will rename it to `notify_operator_count`. For now, document/parse it under its current name.

### Webhook delivery details
- **POST** with `Content-Type: application/json`, `User-Agent: Adros-Monitor/1.0`
- **Signature header:** `X-Adros-Signature: sha256=<hex>`
- **Signature input:** `json.dumps(payload, sort_keys=True).encode("utf-8")` ŌĆö receivers MUST sort keys when verifying
- **Timeout:** 15 seconds (`httpx.AsyncClient(timeout=15.0)`)
- **NO automatic retries** in the current code. 5xx / timeout / network error = logged and dropped. If the receiver is down, that day's alerts are lost unless Adros-side retries are implemented. ŌåÉ **Known gap, documented for awareness.**
- Success = HTTP 2xx only. Any non-2xx is logged as a warning but no retry fires.

### Email fallback
Same run also sends an email via Resend if `monitoring_emails_enabled=true`. Email uses **Haiku 4.5** (`claude-haiku-4-5-20251001`) to generate a friendly 3-5 sentence summary of the issues. Falls back to a plain HTML template if Anthropic key is unavailable or the call fails.

## 8. Creative pipeline (canonical flow for every production creative)

Hard rules baked into the MCP server instructions:
1. **ALWAYS `mode='bold'`** ŌĆö produces editorial / cinematic output (as opposed to `autonomous` which is more literal)
2. **ALWAYS `generator='nano_banana_pro'`** ŌĆö Gemini 2.5 Pro image model, better than nano_banana_2 for ads
3. **ALWAYS `auto_select=True`** on `build_creative_prompt` unless the operator explicitly picks a pattern

### The canonical sequence
```python
# 1. Find strategic match
patterns = match_strategy(
    industry="fitness",
    awareness_stage="problem",
    risk_tolerance="medium"
)

# 2. Compose the prompt (auto-selects best pattern, loads brand + business + persona)
prompt = build_creative_prompt(
    auto_select=True,
    client_name="Acme Travel",  # your client name
    mode="bold"
)
# Internally: loads pattern.blueprint_json + enrichment_metadata
#           + image_generation_prompt (as "REFERENCE VISUAL DIRECTION")
#           + business_context + personas + brand_assets
#           + ad_language from business_context

# 3. Generate the image
image = generate_creative(
    prompt=prompt,
    generator="nano_banana_pro"
)
# Subject to 60s idempotency cache + 20/hr rate limit

# 4. (Optional) QA pass
validate_creative(image_url=image["url"])
```

## 11. Known foot-guns (things that have burned us)

1. **Account mismatch** ŌĆö Adros users are separate accounts. Brand assets uploaded as `test2@adros.ai` won't show for `gst3958@gmail.com`. Always work off `user_id`/`user_token`, never account name.

2. **Client scoping** ŌĆö `client_id=NULL` means solo, `client_id=<uuid>` means agency-specific. The `_client_filter` helper is strict ŌĆö mixing them creates ghost data. `schedule_optimization_review` MUST pass `client_id` explicitly.

3. **MCP `user_token` edge case** ŌĆö if token comes from URL/header context, the param will be empty string. Do NOT guard with `if user_token:` ŌĆö use the context resolver. This bit us in 3 tools in Session 72.

4. **Pattern DB fields hidden from UI** ŌĆö `enrichment_metadata` and `image_generation_prompt` are stripped from the `PatternDetail` Pydantic schema to prevent scraping. They ARE available to MCP calls. Do NOT expose them via any public endpoint.

5. **Lead Gen TOS trap** ŌĆö Meta Lead Gen requires TOS acceptance **per app per page**. Adros being TOS-accepted on one page doesn't carry over. Client must add Adros app ID to their Business Manager as a partner.

6. **Webhook has no retries** ŌĆö see ┬¦7. If OpenClaw's receiver is down at 8am SGT, that day's alerts are dropped. Needs fixing.

7. **Monitor caps at first 3 ad accounts** ŌĆö `user_info.get("meta_ad_account_ids", [])[:3]` and `google_customer_ids[:3]`. Users with > 3 accounts only get the first 3 checked. Deliberate cost control, but worth knowing.

8. **`clients` table has no `is_active` flag** ŌĆö OpenClaw's "which clients are current" question has no answer from the DB today. See integration notes ┬¦3.1.

## 12. Integration contract summary

### Outbound from Adros (what Adros pushes to you)
- **Daily monitoring webhook** ŌĆö 8am SGT, HMAC-signed (`X-Adros-Signature: sha256=<hex>`, sort_keys JSON), v1.1 payload. No retries on failure.

### Inbound to Adros (what you can call)
- **MCP tools** ŌĆö 74 tools via Streamable HTTP at `/mcp`. Auth via `X-User-Token` header.
- **REST endpoints** ŌĆö all under `/api/v1/*`. Auth via Supabase JWT in `Authorization: Bearer`.
- **Pattern endpoints** ŌĆö `/api/v1/tools/*` for direct pattern lookups without MCP overhead.

### Data contracts
- All `.xlsx` output templates are controlled server-side via `get_output_template(type)`
- All workflow playbooks come from `get_workflow_guide(type)`
- All intake questions come from `get_intake_questions(type)`
- **Never hand-roll these schemas on the OpenClaw side.** Ask Adros.

### Authentication
| Surface | Auth |
|---|---|
| MCP tools | `X-User-Token` header OR `user_token` param |
| REST `/api/v1/*` | Supabase JWT `Authorization: Bearer <jwt>` |
| Webhook verification | HMAC-SHA256 of sort_keys JSON body with user's `monitoring_webhook_secret` |
