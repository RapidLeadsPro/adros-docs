# Contract Layer

The contract layer defines **exactly what each specialist must produce and consume** at every lifecycle stage. It is the bridge between Adros (marketing brain + task execution) and whatever is driving the workflow â€” whether that's the native CEO + Command Center or a custom conductor.

---

## How it works

Contracts are keyed by `(stage, task_type)`. Each contract specifies:

| Field | Purpose |
|-------|---------|
| `produce` | Artifacts the specialist must deliver (e.g. `["structure_xlsx", "structure_rationale"]`) |
| `accept` | What the specialist consumes (e.g. `["raw_research", "business_context"]`) |
| `recommended_specialist` | Runtime actor ID: `adros_ceo` / `adros_cmo` / `adros_cd` / `adros_deployer` |
| `review_gates` | Human checkpoints (e.g. "Operator approval") |
| `next_stage` | Where the workflow goes after this task completes |

### Example

```python
("creative", "generate_creatives"): {
    "produce": ["creatives_package", "copy_variants"],
    "accept": ["structure_xlsx", "brand_profile", "business_context"],
    "recommended_specialist": "adros_cd",
    "next_stage": "approved",
    "review_gates": ["Creative Director craft review", "Operator approval"],
}
```

**What happens:**

1. Workflow engine queries `GET /contract/spec?stage=creative&task_type=generate_creatives`
2. Returns the spec above
3. Task is created with `assigned_agent = "adros_cd"` (from `recommended_specialist`)
4. When complete, the specialist calls `POST /contract/writeback` with the produced artifacts
5. State machine advances to `next_stage`

## REST vs MCP

REST is for your Python workers and automation. MCP is for LLM clients calling Adros via natural language. Both use the same contract definitions.

**Dashboard & orchestration REST:** The Contract endpoints above define **deliverable specs** and lifecycle **writeback**. Task inbox, event timelines, approvals, Strategy Workspace (snapshots, masters, stage review), Files, and deploy handoff use **additional** routes on the same `/api/v1` base â€” see [REST â€” Dashboard & Orchestration](/integrations/rest-dashboard).

## Endpoints

### List specs

```
GET /contract/specs
```

Returns all available `(stage, task_type)` specs with their produce/accept/specialist/next_stage/review_gates.

### Get single spec

```
GET /contract/spec?stage=creative&task_type=generate_creatives
```

Returns one spec. Useful when the workflow engine needs to know what's next.

### Writeback (specialist delivers)

```
POST /contract/writeback
Body: { stage, task_type, client_id, artifacts: {...}, notes: "..." }
```

The specialist posts their produced artifacts. The contract service:
1. Validates required artifacts from the `produce` list
2. Stores them in task artifacts + R2
3. Updates client lifecycle state to `next_stage`
4. Logs the writeback as a task event

## Design rationale

**Why contracts?** Without them, each specialist has to know the state machine implicitly â€” and they get it wrong. Contracts make the workflow **deterministic**: given the same (stage, task_type), everyone produces and consumes the same things. This is how you get **traceability** instead of "the AI did something."

**Why runtime actor IDs?** Display names drift. Actor IDs (`adros_ceo`, `adros_cmo`, etc.) are stable, machine-readable, and match the actual task assignment fields in the orchestrator.
